# Phase 5: Programmatic SEO Scale-Up

**Project:** PDF Toolkit
**Estimated session time:** 6–8 hours
**Prerequisites:** Phase 1 deployed (Astro scaffold + sitemap integration + base SEO). Phases 2–4 are useful but not strictly blocking — the SEO build only needs the tool components to exist as Svelte islands; if a given tool isn't built yet, its page becomes a "coming soon" page until the relevant phase lands. Recommended order: ship 1–4 first, then this phase.
**Ships:** Astro Content Collection driving 60+ programmatic landing pages from a single template (12 operations × size variants × India intents × quality modifiers). Full JSON-LD structured data (`SoftwareApplication`, `HowTo`, `FAQPage`, `BreadcrumbList`). Auto-generated sitemap + robots.txt. Internal-link graph between every related tool. Blog scaffold for ongoing content marketing. Lighthouse 95+ across all 60+ pages.

## Context (read me first)

You are building **PDF Toolkit**, a 100% client-side privacy-first PDF utility. Earlier phases have shipped the core tools (merge, split, rotate, reorder, delete, extract, compress, jpg↔pdf, ocr, protect/unlock — 12 total) on Cloudflare Pages from a public MIT-licensed GitHub repo. The Web Worker + Comlink + OPFS architecture handles 300 MB files; the `@jsquash`-based compression pipeline hits exact India size targets (100 KB / 200 KB / 500 KB / 1 MB / 2 MB).

**SEO is ~70% of the growth strategy** for this app (research §8). iLovePDF's documented playbook is "verb + object pages, dense internal linking, ship them fast" — they've parlayed that into 188M monthly visits. We can't beat them on the head terms ("merge pdf"), but the long tail is wide open: "compress pdf to 200kb for ssc cgl", "merge pdf for upsc application", "jpg to pdf high quality no upload", "best free smallpdf alternative", etc. Each long-tail query has documented competitor pages, modest difficulty, and matches the privacy-first wedge.

This phase scales the SEO surface from the ~12 pages of Phases 1–4 to ~60+ unique pages, each driven by a single content-collection entry plus a shared Astro template. Every page renders an existing tool island (Phase 1–4 components) plus targeted content (intro, FAQ, JSON-LD). The same tool widget is reused across all pages — the differentiation lives in the framing copy, internal links, and structured data, not in different code paths.

This is the lowest-engineering, highest-leverage phase. Done right, it sets up months of compounding organic traffic with near-zero ongoing effort beyond writing weekly blog posts (Phase 6 sets that scaffold up — defer the writing).

## Goal of this phase

Stand up an Astro Content Collection–driven landing-page system that emits 60+ statically-generated SEO-optimised pages from JSON entries, each with unique title/description/intro/FAQ + full JSON-LD, full internal link graph, and a complete sitemap.

- [ ] Open Graph + Twitter Card meta tags per page, with a per-tool OG image (use a single template with the tool name swapped in — generate at build via `astro-og-canvas`).
- [ ] `<link rel="canonical">` correctly set on every page.
- [ ] All landing pages link to home + back to the tool hub via breadcrumb.

## Tech stack & dependencies

**New installs (pin versions):**
- `astro-og-canvas@^0.5.7` — generates OG images per-route at build using a satori-like canvas approach. ~free CPU at build, big share-graph win.
- `@astrojs/mdx@^4.0.0` — MDX support in content collections (so intros and FAQs can include components like `<RelatedTools />` inline).

**Already installed (Phase 1+):**
- `astro@^5.5.0`, `@astrojs/sitemap`, `@astrojs/svelte`, etc.

**Rationale (research §6, §8):**
- Content collections + dynamic routes is Astro's native "programmatic SEO" pattern; emits pure static HTML at build, ~0 KB JS, perfect for ranking.
- MDX (vs Markdown) lets you embed `<CompressTool>` directly in the body of any landing page entry, removing the need for separate page files.
- OG images materially impact CTR from Twitter/X, LinkedIn, Slack — `astro-og-canvas` is the lowest-effort way to ship per-page OG images.

## Step-by-step tasks

1. **Install deps (~5 min):** `npm i astro-og-canvas @astrojs/mdx && npx astro add mdx`.
2. **Define the collection schema** in `src/content/config.ts` — a Zod schema for `landing` and `blog`. The landing entry includes `tool` (an enum of all 12 tool keys), optional `presetParam`, an `intentTag`, and a `relatedSlugs` array.
3. **Build `Landing.astro` layout:** extends `Base.astro`, adds a `<Breadcrumbs>` slot, a hero with H1, a `<slot />` for MDX content, a `<RelatedTools>` block, and JSON-LD emission via `<JsonLd>`, `<FaqSchema>`, `<HowToSchema>`.
4. **Build `ToolLoader.astro`:** accepts a `tool` prop and switches to render the right Svelte island with `client:visible`. This is the glue between content data and tool components.
5. **Build `src/pages/[...slug].astro`:**
   - `export async function getStaticPaths()` reads `getCollection('landing')` and returns `{ params: { slug }, props: { entry } }` for each.
   - The body renders `<Landing>` populated from `entry.data`, with `<entry.Content />` (MDX) inside.
   - Sets canonical to `site + '/' + slug`.
6. **Author the first content entry** as a template — `src/content/landing/compress-pdf-to-200kb.mdx`. Frontmatter has all the schema fields; body is the 300-word intro + 4 FAQ entries + an embedded `<CompressTool client:visible targetParam="200kb" />`.
7. **Bulk-author the 60+ entries.** This is the bulk of the phase's time. Don't AI-spin — write each intro genuinely (sentence variation, real use case, real example portal name). Drop pasting source: a Google sheet or just a markdown checklist. Each entry: ~5 minutes if you have a content template. **Honest prioritisation:** if you run out of time, ship the 30 highest-volume ones first (the compression sizes + India intent pages are the highest-ranking long-tail).
   - **Categories to author (target counts):**
     - Operations (hub pages): 12 — these already exist; port to MDX.
     - Compression sizes: 10.
     - Compression for-X India intents: 10.
     - JPG→PDF for-X: 5 (Aadhaar, PAN, college, exam form, KYC).
     - JPG→PDF size variants: 3 (`-less-than-100kb`, `-less-than-200kb`, `-less-than-500kb`).
     - Quality / use modifiers: 8.
     - Alternative-to comparisons: 5.
     - Long-tail combinations: 7+ (e.g., `merge-pdf-for-upsc`, `pdf-to-jpg-high-quality-no-upload`).
8. **Wire `JsonLd.astro`:** accepts a JS object, JSON.stringifies, emits inside a `<script type="application/ld+json">`. Single component reused by all the schema-specific wrappers.
9. **Author `FaqSchema.astro`:** receives `faq: { q, a }[]` and emits `FAQPage` JSON-LD.
10. **Author `HowToSchema.astro`:** for procedural pages (e.g., a how-to-merge page), emits `HowTo` JSON-LD with steps "Drop your files / Reorder / Click Merge / Download." 5 generic steps per tool, parametrised.
11. **Author `Breadcrumbs.astro`:** visual breadcrumb plus `BreadcrumbList` JSON-LD; reads slug to compute trail (Home → Compress PDF → 200 KB).
12. **Author `RelatedTools.astro`:** takes `relatedSlugs[]` from the entry, resolves them to titles via `getCollection`, renders as a grid.
13. **Configure OG images:** `src/lib/og.ts` exports a function returning an `OGImageRoute` per landing page; routes auto-emit `/og/<slug>.png` at build. Use brand colours, tool name, "100% in your browser" tagline.
14. **Add the OG image URL to `Base.astro`** meta tags (`og:image`, `twitter:image`) sourced from the entry slug.
15. **Generate the human-readable sitemap** at `/sitemap.html.astro` — iterates `getCollection('landing')` and `getCollection('blog')` and lists all pages grouped by category.
16. **Set up the blog:** content schema for blog with `title`, `description`, `pubDate`, `targetSlug` (the landing page the post supports). Author 3 seed posts; each ends with "Use the tool: [link]".
17. **`/blog/index.astro`:** lists posts reverse-chronologically with summary + read time.
18. **Run a build (`npm run build`).** Confirm `dist/` contains all expected HTML files. Spot-check 5 random landing pages by opening the built HTML.
19. **Sitemap verification:** open `dist/sitemap-index.xml` (auto by `@astrojs/sitemap`), confirm every slug listed. Submit URL to Google Search Console + Bing Webmaster Tools (this is part of Phase 6 polish; just verify the file is correct now).
20. **Internal-linking sweep:** every landing page links to (a) at least 2 other landing pages via `RelatedTools`, (b) the home page, (c) the relevant tool hub. Run a quick script (`scripts/check-links.mjs`) to validate.
21. **Lighthouse sweep:** sample 5 random landing pages on mobile. All ≥ 95. Common failure modes at scale: missing `alt` on OG fallback, missing `<meta description>` on a typo'd entry — fix the schema to make these required and re-validate Zod at build.
22. **Update README:** mention the 60+ programmatic landing pages and link to the human sitemap.
23. **Commit + deploy.** Submit sitemap to Google Search Console after deploy (creates account if not already done).

## Key code patterns

**`src/content/config.ts`:**
```ts
import { defineCollection, z } from 'astro:content';

const TOOLS = ['merge','split','rotate','reorder','delete','extract','compress','jpg-to-pdf','pdf-to-jpg','ocr','protect','unlock'] as const;

const landing = defineCollection({
  type: 'content',
  schema: z.object({
    h1: z.string(),
    metaTitle: z.string().max(70),
    metaDescription: z.string().min(120).max(160),
    tool: z.enum(TOOLS),
    presetParam: z.string().optional(),
    intentTag: z.enum(['operation','size','india-govt','quality','alternative','combination']).optional(),
    relatedSlugs: z.array(z.string()).default([]),
    faq: z.array(z.object({ q: z.string(), a: z.string() })).min(3),
    publishedAt: z.string().optional(),
  }),
});

const blog = defineCollection({
  type: 'content',
  schema: z.object({
    title: z.string(),
    description: z.string(),
    pubDate: z.date(),
    targetSlug: z.string(),
    author: z.string().default('PDF Toolkit'),
  }),
});

export const collections = { landing, blog };
```

**`src/pages/[...slug].astro`:**
```astro
---
import { getCollection } from 'astro:content';
import Landing from '@/layouts/Landing.astro';
import ToolLoader from '@/components/ToolLoader.astro';
import FaqSchema from '@/components/seo/FaqSchema.astro';
import HowToSchema from '@/components/seo/HowToSchema.astro';

export async function getStaticPaths() {
  const entries = await getCollection('landing');
  return entries.map(entry => ({ params: { slug: entry.slug }, props: { entry } }));
}

const { entry } = Astro.props;
const { Content } = await entry.render();
const { h1, metaTitle, metaDescription, tool, presetParam, faq, relatedSlugs } = entry.data;
---
<Landing title={metaTitle} description={metaDescription} {h1} {relatedSlugs} canonical={`/${entry.slug}`}>
  <Content />
  <ToolLoader {tool} {presetParam} />
  <section class="faq">
    <h2>Frequently asked questions</h2>
    {faq.map(({q,a}) => <details><summary>{q}</summary><p>{a}</p></details>)}
  </section>
  <FaqSchema {faq} slot="schema" />
  <HowToSchema {tool} slot="schema" />
</Landing>
```

## Gotchas (from research)
- **Don't AI-spin content.** Google's 2024–2025 helpful-content updates demote thin spun-feeling text. Each landing-page intro must reference a real, specific use (e.g., "SSC CGL 2026 application accepts up to 200 KB photo+sig PDFs"), not "compress your PDF to 200 KB easily!"
- **Canonical tags matter.** Without `<link rel=canonical>`, near-duplicate pages (e.g., the size-variants of compress) compete and rank-cannibalise. Always set canonical to the page's own URL.
- **Internal linking is the #1 multiplier.** Stirling-PDF + iLovePDF rank as well as they do partly because every page links to 5–10 related tools (research §8). Skipping the `RelatedTools` block is the single most expensive shortcut.
- **`@astrojs/sitemap` does NOT include dynamic routes from content collections by default** if the routes use `[...slug].astro` with `getStaticPaths` — actually it does, but only if `getStaticPaths` is reachable at build. Run `npm run build` once and check `dist/sitemap-0.xml` lists every slug; if missing, add `customPages` config.
- **Cloudflare Pages can serve 25k routes** without performance degradation, so the 60-page target is comfortably within limits. Don't worry about scale here.
- **OG image generation can blow up build time** if you have 200 pages × 1 image each. `astro-og-canvas` is fast (~50 ms each) but it adds up. Acceptable at 60 pages; cap at ~200 in the future.
- **Lighthouse SEO score is sensitive to `<meta name=robots>` being absent or wrong.** Add `index, follow` explicitly on landing pages; never leave it implicit.
- **MDX components inside MDX content must be imported in the MDX file OR globally registered.** Easier path: only use components defined in the layout (which renders `<Content />`); pass complex bits as props.
- **JSON-LD validation:** Google's Rich Results Test (search.google.com/test/rich-results) flags missing `image` on `SoftwareApplication`; include a default OG image URL there to avoid the warning.
- **Avoid keyword stuffing in MDX intros.** Each page targets one primary keyword in H1 + meta-title + first paragraph + once more in the body. More than that triggers spam detection.
- **Don't put `noindex` on `/blog/index.astro`** by accident if you copy Astro starter blog templates that paginate — they sometimes mark non-first pages as noindex.
- **Blog posts should link to landing pages, not the other way around** — landing pages are the conversion target; blog is the discovery channel.
- **PDF→Word remains out-of-scope** (research §6). Don't author a `/pdf-to-word` landing page; you'd disappoint everyone who landed.
